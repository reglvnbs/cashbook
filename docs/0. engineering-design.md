# 账本工程设计要求

本规范定义账本项目的技术选型、代码组织、数据访问、接口和代码质量要求。

## 技术栈

| 领域 | 选型 | 要求 |
| --- | --- | --- |
| 运行环境 | Python 3.13 | 生产、开发和测试使用同一主版本 |
| Web 框架 | Flask 3.x | 使用应用工厂创建应用 |
| 页面模板 | Jinja2 | 服务端渲染页面基础结构 |
| 前端 | 原生 HTML、CSS、JavaScript | 不引入大型前端框架和打包工具 |
| 数据库 | SQLite | 直接使用参数化 SQL，不引入 ORM |
| AI 服务 | DeepSeek API | API Key 仅从环境变量读取 |
| 生产服务 | Gunicorn | `wsgi.py` 作为启动入口 |
| 容器 | Docker | 数据通过宿主机目录或 Volume 持久化 |
| 测试 | pytest | 由 `test.sh` 统一管理测试环境 |

依赖分别记录在 `requirements.txt` 和 `requirements-dev.txt`。生产依赖不包含只用于测试、格式检查或调试的工具。

## 分层结构

代码调用方向固定为：

```text
Route → Service → Repository → Database
```

| 层 | 职责 |
| --- | --- |
| Route | 接收请求、读取参数、调用 Service、返回页面或 JSON |
| Service | 执行业务校验、统计计算、事务组合和 DeepSeek 调用 |
| Repository | 执行参数化 SQL，并把数据库记录转换为业务数据 |
| Database | 管理 SQLite 连接、事务、外键和初始化 |

约束：

- Route 不直接执行 SQL，也不包含统计计算。
- Repository 不处理 HTTP 请求、表单文案和第三方 API。
- DeepSeek 调用放在 Service 层，识别阶段不写数据库。
- 一次确认多笔 AI 流水时使用同一个数据库事务。
- 通用函数只有在不依赖具体业务时才放入 `utils.py`。

## 项目目录

```text
app/
├── routes/          页面路由和 JSON API
├── services/        业务逻辑和外部服务调用
├── repositories/    数据访问
├── templates/       Jinja2 模板
├── static/          CSS、JavaScript、SVG 等静态资源
├── config.py        配置加载与校验
├── database.py      SQLite 连接和事务
└── schema.sql       初始表结构、索引和分类数据

data/                正式数据库、日志和运行数据
build/test/          隔离的测试与本地运行环境
docs/                需求和设计文档
tests/               pytest 测试代码
```

源码目录只保存源码和静态资源。数据库、日志、虚拟环境、缓存和测试输出不得出现在源码目录中。

## 数据与 SQLite

- 正式数据保存到配置指定的 `data/` 路径。
- 应用启动时创建缺失的数据目录，但不自动删除或覆盖已有数据库。
- 每个 SQLite 连接启用外键约束。
- 所有 SQL 使用占位符传参，不拼接用户输入。
- 多条相关写操作使用显式事务，失败时整体回滚。
- `schema.sql` 只定义数据库结构、索引和固定分类，不包含演示流水。

## HTTP 与接口

- 页面路由 `/`、`/transactions` 和 `/budget` 返回 HTML。
- 应用不提供用户注册、登录和多用户权限；浏览器写请求使用 CSRF 防护。
- 业务接口统一放在 `/api` 下，使用 UTF-8 JSON。
- 请求和响应中的金额以元为单位，使用两位小数字符串；数据库内部仍使用整数分。
- 日期使用 `YYYY-MM-DD`，月份使用 `YYYY-MM`，时间使用带时区的 ISO 8601 字符串。
- 所有写接口执行服务端校验，不能只依赖浏览器校验。

### 通用响应

成功响应统一放在 `data` 中：

```json
{ "data": {} }
```

失败响应统一使用以下结构，`fields` 只在字段校验失败时出现：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "请检查填写内容",
    "fields": { "amount": "金额必须大于零" }
  }
}
```

| 状态码 | 使用场景 |
| --- | --- |
| `200` | 查询、修改和删除成功 |
| `201` | 新增一笔或批量新增成功 |
| `400` | 参数格式或业务校验失败 |
| `401` | Bearer Token 缺失或错误 |
| `403` | CSRF 校验失败 |
| `404` | 流水等资源不存在 |
| `503` | 自动化接口或 AI 功能未配置、外部 AI 服务不可用 |
| `500` | 未处理的服务端错误 |

稳定错误代码包括 `VALIDATION_ERROR`、`NOT_FOUND`、`AUTH_FAILED`、`CSRF_FAILED`、`AUTOMATION_DISABLED`、`AI_NOT_CONFIGURED`、`AI_SERVICE_ERROR`、`AI_INVALID_RESPONSE` 和 `INTERNAL_ERROR`。错误响应不得包含堆栈、SQL、文件路径、密钥或第三方完整响应。

### 接口清单

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/health` | 检查应用和数据库状态 |
| `GET` | `/api/categories` | 读取固定分类 |
| `GET` | `/api/overview` | 读取指定日期范围的总览 |
| `GET` | `/api/transactions` | 筛选并分页读取流水 |
| `GET` | `/api/transactions/{id}` | 读取一笔流水 |
| `POST` | `/api/transactions` | 新增一笔流水 |
| `PUT` | `/api/transactions/{id}` | 修改一笔流水 |
| `DELETE` | `/api/transactions/{id}` | 删除一笔流水 |
| `POST` | `/api/transactions/batch` | 原子批量新增 AI 草稿 |
| `POST` | `/api/ai/parse` | 将自然语言解析为流水草稿 |
| `GET` | `/api/budgets/{month}` | 读取一个月的预算状态 |
| `PUT` | `/api/budgets/{month}` | 一次保存整月预算 |

不提供分类写接口、预算删除接口和配置读取接口。

`GET /api/health` 成功时返回 `{"data":{"status":"ok","database":"ok"}}`。数据库不可用时返回 HTTP `503`，健康检查不执行任何业务写入。

### 分类

`GET /api/categories` 在 `data` 中返回按编号升序排列的分类数组。可以使用 `transaction_type=income` 或 `transaction_type=expense` 筛选。

分类对象格式为：

```json
{
  "id": 1,
  "name": "餐饮",
  "transaction_type": "expense",
  "color": "#C77956"
}
```

### 流水

流水对象格式为：

```json
{
  "id": 123,
  "transaction_type": "expense",
  "amount": "32.00",
  "category": { "id": 1, "name": "餐饮", "color": "#C77956" },
  "occurred_on": "2026-07-19",
  "note": "午饭",
  "created_at": "2026-07-19T12:30:00+08:00",
  "updated_at": "2026-07-19T12:30:00+08:00"
}
```

`GET /api/transactions` 支持以下查询参数：

| 参数 | 规则 |
| --- | --- |
| `keyword` | 匹配备注或分类名称，空白值忽略 |
| `start_date`、`end_date` | 选填，包含边界；只填写一个时作为单侧范围 |
| `types` | 逗号分隔的 `income`、`expense`，同一参数内使用“或” |
| `category_ids` | 逗号分隔的分类编号，同一参数内使用“或” |
| `page` | 从 `1` 开始，默认 `1` |

不同筛选参数之间使用“且”。不提供日期参数时读取全部流水。结果按发生日期倒序、流水编号倒序排列，每页数量来自配置。分页响应格式为：

```json
{
  "data": {
    "items": [],
    "pagination": { "page": 1, "page_size": 20, "total_items": 0, "total_pages": 0 }
  }
}
```

新增和修改使用同一字段结构；`PUT` 需要提交全部字段：

```json
{
  "transaction_type": "expense",
  "amount": "3200.00",
  "category_id": 4,
  "occurred_on": "2026-07-01",
  "note": "房租"
}
```

- `amount` 必须大于零且最多两位小数，不接受自动四舍五入。
- `category_id` 必须存在，并与收支类型一致。
- `occurred_on` 不能晚于今天。
- `note` 缺省为空字符串，去除首尾空白后最多 200 个字符。
- 删除成功返回 `{"data":{"id":123}}`；不存在的编号返回 `404`。

`POST /api/transactions/batch` 的请求格式为 `{"transactions":[...]}`，每项使用普通流水新增字段，一次允许 1 至 20 笔。全部校验通过后在一个事务中写入；任意一笔失败时不保存任何流水。成功返回完整流水对象数组。

### 自动新增流水

`POST /api/transactions` 同时供页面和定时脚本使用，两种调用进入同一个 `TransactionService`。

- 页面请求使用 `X-CSRF-Token`。
- 自动化请求使用 `Authorization: Bearer <CASHBOOK_API_TOKEN>`，不要求 CSRF Token。
- Bearer Token 只允许调用单笔新增接口，不能查询、批量新增、修改或删除数据。
- 未配置 `CASHBOOK_API_TOKEN` 时，携带 Bearer Token 的请求返回 `503 AUTOMATION_DISABLED`，页面记账不受影响。
- Token 错误返回 `401 AUTH_FAILED`，日志不得记录 Token 或完整请求头。

### 总览

`GET /api/overview` 接受成对的 `start_date` 和 `end_date`。两者都不提供时，默认使用本月第一天至今天；只提供其中一个、开始晚于结束或结束晚于今天时返回 `400`。

返回内容包括：

```json
{
  "data": {
    "date_range": { "start_date": "2026-07-01", "end_date": "2026-07-19" },
    "summary": { "income": "10000.00", "expense": "3200.00", "balance": "6800.00" },
    "expense_categories": [
      {
        "category": { "id": 4, "name": "居住", "color": "#8173A2" },
        "amount": "3200.00",
        "percentage": 100.0
      }
    ],
    "budget_summary": null
  }
}
```

支出分类按金额倒序、分类编号升序排列。没有支出时返回空数组。日期范围跨月时 `budget_summary` 为 `null`；属于同一个自然月时，其结构与预算查询响应相同，预算金额取该月设置，已使用金额只统计所选日期范围内的支出。

### 预算

`GET /api/budgets/{month}` 返回总预算和全部支出分类预算。每项包含 `amount`、`used`、`remaining` 和 `usage_percent`；未设置预算时 `amount`、`remaining` 和 `usage_percent` 为 `null`。`used` 始终统计该自然月的全部支出。

```json
{
  "data": {
    "month": "2026-07",
    "total_budget": {
      "amount": "8000.00",
      "used": "3200.00",
      "remaining": "4800.00",
      "usage_percent": 40.0
    },
    "category_budgets": [
      {
        "category": { "id": 1, "name": "餐饮", "color": "#C77956" },
        "amount": null,
        "used": "600.00",
        "remaining": null,
        "usage_percent": null
      }
    ]
  }
}
```

分类预算数组始终包含全部支出分类并按编号排序。`remaining` 可以为负数，`usage_percent` 保留两位小数且可以超过 100。

`PUT /api/budgets/{month}` 一次提交整页状态：

```json
{
  "total_budget": "8000.00",
  "category_budgets": [
    { "category_id": 1, "amount": "1500.00" },
    { "category_id": 2, "amount": null }
  ]
}
```

- `month` 必须使用 `YYYY-MM`，允许设置未来月份。
- 请求必须包含所有支出分类，每个分类只能出现一次。
- 金额必须大于零；`null` 表示未设置，也用于清除已有金额。
- 保存是一个完整事务，失败时保留修改前的全部预算。
- 保存成功返回与预算查询相同的完整状态。

### AI 解析与返回格式

`POST /api/ai/parse` 使用 CSRF 防护，请求格式为：

```json
{ "text": "昨天午饭32元，地铁4元" }
```

`text` 去除首尾空白后长度为 1 至 1000 个字符。解析阶段不写数据库，成功后返回可编辑草稿：

```json
{
  "data": {
    "drafts": [
      {
        "transaction_type": "expense",
        "amount": "32.00",
        "category_id": 1,
        "occurred_on": "2026-07-18",
        "note": "午饭"
      },
      {
        "transaction_type": "expense",
        "amount": "4.00",
        "category_id": 2,
        "occurred_on": "2026-07-18",
        "note": "地铁"
      }
    ]
  }
}
```

发送给 DeepSeek 的提示必须包含今天的日期、固定分类编号及收支类型，并要求模型只返回以下 JSON，不得添加 Markdown、解释或其他字段：

```json
{
  "transactions": [
    {
      "transaction_type": "expense",
      "amount": "32.00",
      "category_id": 1,
      "occurred_on": "2026-07-18",
      "note": "午饭"
    }
  ]
}
```

- 模型必须返回 1 至 20 笔，每笔包含全部五个字段；备注可以为空字符串。
- 相对日期以应用认定的今天为基准；没有日期时使用今天，未来日期无效。
- Service 层按普通流水规则重新校验模型结果，不能信任模型输出。
- JSON 无法解析、字段缺失或任意草稿无效时，返回 `503 AI_INVALID_RESPONSE`，不返回部分结果。
- 未配置 API Key 时返回 `503 AI_NOT_CONFIGURED`；超时或外部服务失败时返回 `503 AI_SERVICE_ERROR`。
- 用户确认时调用批量新增接口；用户原文、模型原始响应和未确认草稿不写入数据库或日志。

## 代码质量

- 优先选择直接、易读的实现，不为简单功能增加抽象层。
- 函数保持单一职责，名称表达业务含义。
- 注释解释设计原因、边界和异常情况，不复述代码本身。
- 公共函数和跨层数据结构使用必要的类型标注。
- 金额、日期、分页和错误格式由公共实现统一处理。
- 前端保持原生模块化代码，避免全局变量和重复 DOM 逻辑。
- 删除无用代码和临时调试内容，不保留注释掉的大段实现。
- 新功能不得扩展出数据库设计和界面设计之外的业务概念。
